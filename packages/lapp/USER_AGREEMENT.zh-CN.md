- “Vault”指兼容 LAPP 软件用于保存 Profile 所引用凭据的、可选的当前用户 OS

## 3. LAPP 做什么、不做什么

LAPP 是本地 Provider 与模型 Registry。应用读取 Profile，解析模型、endpoint 和
凭据，然后直接与选中的上游 Provider 通信。

LAPP 不是：

- AI 模型或上游 Provider；
- proxy、gateway、防火墙、凭据隔离 broker 或安全沙箱；
- 对同一 OS 用户权限下不同应用的隔离机制或恶意软件防护；
- 身份、访问控制、额度、计费、退款或审计系统；
- 内容审核、事实核验、法律合规或人工复核服务；
- 备份、高可用、多用户或生产级 secret 管理系统。

兼容 LAPP 软件可以提供 Vault 支持的静态加密凭据存储。Vault 不会阻止同一 OS
用户下的兼容应用取得明文凭据并直连 Provider。发行方可以在 LAPP 规范之外增加
功能。该等功能由发行方负责，并应在需要时另行披露。

## 4. 你明确知悉的主要风险

接受本协议即表示你知悉以下全部风险：

1. **凭据访问。** 能够解析可用 Profile 或共享 Vault 引用的应用，可以取得并
   使用其中引用的 Provider 凭据。LAPP v1 假定同一 OS 用户下运行的应用可信；
   Vault 不是逐应用访问边界，也不是不可导出凭据机制。
2. **目的地控制。** Profile 同时控制凭据引用和发送目的地。恶意或错误的 Profile
   可能把凭据与内容发送给非预期 Provider。校验可以减少部分错误，但不能证明所
   配置的目的地可信。
3. **第三方数据处理。** Prompt、消息、system instruction、对话历史、工具定义和
   结果，以及其他提交内容会直接发送给选中的 Provider。Provider 的保存、训练、
   审核、所在地、跨境传输和隐私规则可能适用。
4. **费用与账号影响。** 真实 Provider 请求会消耗你的账号额度，并可能产生费用、
   税费、限流、封禁或其他账号后果。LAPP 不执行消费上限。
5. **不可靠的 AI 输出。** 模型输出可能错误、虚构、不完整、过时、带偏见、有害、
   不安全或受法律限制，也可能包含错误代码、虚假引用、个人信息或侵犯第三方权利
   的内容。
6. **工具和自动化后果。** 如果应用启用工具或自动操作，模型可能触发修改文件、
   调用服务、发送消息、购买商品或产生其他副作用的 handler。参数结构通过校验，
   不代表行为本身安全或符合你的真实意图。
7. **本机与开发环境边界。** LAPP v1 主要面向个人本机和开发使用，不能替代生产
   环境所需的 workload identity、托管 secret store、受限授权、轮换、审计或多
   租户隔离。

## 5. 凭据与 Profile 安全

Plaintext 凭据会留在磁盘上的 Profile 文件中。环境变量或 Vault 引用可以避免把值
写入 Profile，但解析后的值仍会进入应用进程内存。Vault 记录保存在当前用户的 OS
凭据库中，同一用户下的兼容应用可能读取。Query 认证会把凭据放入请求 URL，该 URL
可能进入 Provider、proxy、诊断或请求日志。

官方 SDK 会把 Vault 记录绑定到 Provider ID、标准化 origin 和认证形态。这可以
减少 Profile 改动后意外转发凭据的风险，但恶意同用户应用可以绕过 SDK 检查并直接
访问共享记录。Vault 记录缺失、损坏、被拒绝或绑定不匹配时会直接失败，不会回退到
plaintext、环境变量或文件。

LAPP 不同步或备份 Vault 记录。重装 OS、重置账户或凭据库、更换设备，或者丢失上游
key，都可能使凭据不可用。你仍需在 Provider 侧保留独立的恢复或轮换方式。

传输与校验措施无法保护已经失陷的设备、恶意本地应用、不可信 Profile root、恶意
配置的 endpoint、屏幕捕获、shell history、备份，或 LAPP 软件之外产生的日志。

你应负责：

- 只安装你愿意授予 Provider 使用权的可信应用；
- 使用前检查 Profile root、Provider origin、认证类型和模型；
- 优先使用 Vault 或外部管理的环境变量引用，不使用 plaintext 凭据；
- 在 Provider 支持时使用受限、最小权限的凭据和预算上限；
- 使用适当的 OS 权限保护 Profile、Vault 记录、环境变量、进程输出、日志和备份；
- 怀疑泄露后及时轮换或撤销凭据；以及
- 把任何 reveal 后的连接信息或凭据视为 secret。

卸载 LAPP 软件不一定会删除 Profile、Vault 记录、环境变量、shell 输出、Provider
侧数据或凭据。需要时，你必须另行删除残留数据并撤销凭据。

## 6. 数据、隐私与第三方 Provider

LAPP 官方架构不运行一个居中的 LAPP 中间服务。这不表示你提交的数据会留在设备
本地：选中的 Provider 会收到完成请求所需的内容与凭据。发行方或第三方应用也可能
在 LAPP 规范之外处理数据。

提交个人、机密、受监管、出口管制或第三方数据前，应检查发行方与 Provider 当前的
条款和隐私告知，包括保存期限、模型训练使用、人工审核、数据所在地、sub-processor、
安全、删除和跨境传输。你必须取得提交这些数据所需的权限、告知、同意和其他法律
依据。

应尽量减少提交数据。不要把凭据或非必要个人信息放入 Prompt、模型 metadata、工具
结果、日志或 bug report。

本协议不是隐私政策。发行方如果收集或控制个人信息，必须在法律要求时提供自身合法
充分的隐私告知和同意机制。

## 7. AI 输出、工具与重要决策

应把全部模型输出视为不可信输入。依赖重要事实、引用、计算、代码、命令或建议前，
必须通过独立来源和合格人员进行核验。

LAPP 软件未被设计或验证为医疗、法律、金融、就业、信贷、教育、公共安全、关键
基础设施，或其他可能实质影响个人权利、健康、安全、生计和基本服务获取的决定的
唯一依据。在这些场景使用时，应实施适当的专家复核、测试、人工监督、授权、记录和
申诉机制。

如果启用工具或自动操作，应使用最小权限 handler，并在适当场景使用沙箱、预览、
重要操作确认、消费上限和审计日志。注册并运行 handler 的个人或应用仍对其后果
负责。

输出是否可以被拥有、使用、发布或受保护，取决于 Provider 条款与适用法律。任何
输出都不保证唯一、可受著作权保护、准确、保密或不侵权。

## 8. Provider 账号、费用与可用性

你应负责 Provider 账号、订阅、凭据、用量、税费、地域限制、acceptable-use 规则
和请求产生的费用。Chat、streaming、重试、测试、工具循环、模型发现及其他操作都
可能发出真实网络请求。

Provider 的价格、模型、协议、限制和可用性可能变化，LAPP 不会另行通知。Provider
可以依据自身条款拒绝、审核、保存、延迟、限流或停止请求。LAPP 不承诺某个 Provider
或模型会持续可用，也不对 Provider 收取的费用退款。

## 9. 本地数据与配置完整性

本地模型目录是权威目录。远端刷新只能显式执行且只追加，因此上游已经删除的模型仍
可能保留，直到你主动删除。使用前应核对模型身份、能力、价格和可用性。

LAPP v1 假定只有一个 writer，不提供 Profile 级锁、多文件事务、合并解决或自动
备份。多个 writer、变更中断、手工编辑、存储故障或恶意本地篡改可能造成不一致或
数据丢失。应保留适当备份，避免同时写入，编辑后进行校验，并在 apply 前审查变更。

本地开发可以使用 loopback HTTP，但它没有 TLS 加密。不要把无需认证的本地模型
服务暴露给不可信用户或网络。
